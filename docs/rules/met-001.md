# MET-001 — metric attributes have bound cardinality

**Impact:** Important (weight 30) · **Target:** metric · **Source:** spec ·
**Confidence:** poller-verifiable (`mimir_label_cardinality`)

Attribute keys on metrics, aggregated by metric name, MUST have fewer than
10,000 unique values within a 1-hour window (spec default; tune via
`params.max_cardinality`). High-cardinality attributes explode active series —
the dominant cost and performance failure mode of Prometheus-family backends.

**Detection.** The sampled stream feeds one HyperLogLog sketch per
(service, metric, attribute) — estimates, never exact sets, with a hard cap on
tracked pairs (overflow is reported in the report notes, not silently
dropped). The Mimir poller re-checks flagged pairs against the cardinality API
(`distinct_label_values_count`), which sees the unsampled stream; that verdict
upgrades the finding to `verified`. Requires `cardinality_analysis_enabled:
true` on the tenant (the kind dev cluster enables it).

**Remediation.** Drop or hash the attribute at the collector (template
`high-cardinality-attribute`, ships later in Phase 1); typical offenders:
user IDs, request IDs, raw URLs.
